Many of the previous hurdles from my trial-run webpage, WhatAKerfuffle.ml prepared me for the limitations of hosting this webpage for free. In the writeup I explain how Github Pages and Freenom.com can be used to host a webpage for free with a custom domain, and the only limitations being open source and not https secured. Since this is an open project and I do not collect nor store any user information, these limitations were acceptable costs for a free webpage.

In addition, I set out to improve upon the mistakes previously made. The most pressing being an organized file structure. While the trial webpage wasn’t terrible, it was a very manual system with lots of refactoring. This time, I sought out for a static webpage file management system. Luckily, I stumbled upon Github's Jekyll static webpage generator.



"Jekyll is a simple, blog-aware, static site generator perfect for personal, project, or organization sites. Think of it like a file-based CMS, without all the complexity. Jekyll takes your content, renders Markdown and Liquid templates, and spits out a complete, static website ready to be served by Apache, Nginx or another web server. Jekyll is the engine behind GitHub Pages, which you can use to host sites right from your GitHub repositories"

It took a couple of days of hammering through examples and trial-and-error processes, but by the end I had a comprehensive understanding of the file system, variables, imports, and site generation. This system is fantastic and enabled this webpage's file system to be less manual and better organized.

Trial-run Webpage vs Portfolio Webpage

As seen above, the portfolio webpage has all subpages neatly organized. What's not shown here is how the portfolio webpage utilizes only one CSS file - which is optimal in terms of bandwidth, while the other has separate CSS files for each page - which aids in personal organization. Jekyll allows imports of numerous CSS files, and will compile all into one CSS file when building the webpage. This enables the user to be fully organized on their repository, while also being optimized for bandwidth use. With these structures in place, it was time for actual web design.

Before coding anything, a well thought-out design must be in place. This reduces the time spent adjusting code and leads to overall better structured source code. While the end result may diverge from the original design, the basic design and structure aids in predicting future functionality needs between pages. See the initial design below.

Multiple small adjustments were made to the design since starting the project such as: adding a logo, removing personal image, swapping the order of Skill Sets and Sample Projects, etc. All of these small adjustments lead to a more visually clean and inciting experience for the user. Most of these adjustments were suggestions from one of my professors, Dr. Armstrong. Her feedback was invaluable.

There are a few big-ticket features with this webpage. First being the use of Liquid and YAML headers. These headers reduce the need for repeated code by allowing the user to declare variables accessed through Liquid tags. For example, project writeup pages, like this one, share the following YAML front matter. I have added comments for clarity.

{% highlight c %} --- layout: project title: projectFolder: dated: // optional tags: description: // project's thumbnail text img: // project's thumbnail and abstract image abstract: end: // if there is no ending image end-left: // used for text if there is an ending image end-right: // used for img --- {% endhighlight %}

Following this header is the page's main body content - what you're reading through right now. This allows the rest of the HTML to sit in a layout HTML file titled project.html which uses the Liquid variables given here to build the page. This greatly reduces the need to repeat snippets of code, such as the div declaration. One adjustment to the project.html will automatically be reflected on all the pages that use the layout, which allows for speedy reformatting when needed.

This is not the only used of Liquid for this webpage. Liquid is also used to generate the tag system used. Should one of the tags listed at the beginning of this page be clicked, it will bring the user to a page that lists all projects that contain that tag. In addition, on the homepage under Skill Sets the "See example" text will bring the user to the tag page. Liquid is used to gather all the pages listed with the desired tag, and thus eliminates the need to manually fill and update every tag page available.

Liquid was an easy language to learn, and offers many complex behaviors that reduces a creator's manual work. In a similar vein, SASS also enhanced my ability to stay organized and without repeated code. SASS is a CSS extension that allows use of variables, nested rules, imports, and much more. SASS is not only easier to read, it also enables variables which reduce the need to scatter common values throughout the code. For example, text color, main accent color, and link color will always be the same throughout all pages, so a variable can be assigned to each and used in place of the hex color. This enables the user to only make one change to the variable if he or she ever wants to change the color, rather than having to dig through their code and find all instances.

{% highlight sass %} $textColor: {{ site.data.layout.textColor }} $mainAccent: {{ site.data.layout.mainAccent }} $mainAccent-light: lighten($mainAccent, 25%) $mainAccent-dark: saturate(darken($mainAccent, 10%), 20%) $grayAccent: {{ site.data.layout.grayAccent }} $grayAccent-light: lighten($grayAccent, 25%) {% endhighlight %}

As seen above, I declare six different colors that are used throughout the webpage. Say I want to change the main accent color from blue to green, it will only take the adjustment of $mainAccent for the whole webpage to follow suite. This kind of capability is not available in CSS, yet SASS is close enough to the the language it is easy to pickup and learn.
